<objective>
Add an optional `rr_timestamps_s: Option<Vec<f64>>` field to `HrvInput` and make RMSSD computation in `goose_hrv_v0` BLE-gap-aware: when timestamps are present, any gap > 3.0 s between consecutive RR timestamps starts a new segment, and successive differences are only computed within a segment (cross-boundary diffs are excluded). This stops BLE dropouts from inflating overnight RMSSD (ALG-HRV-01).

Purpose: A 4 s BLE gap currently produces a spurious large successive difference that inflates RMSSD; segment-aware computation excludes it.
Output: `HrvInput.rr_timestamps_s` field; a testable `segment_rr_by_gaps` free function; segment-aware RMSSD in `goose_hrv_v0`; tests proving gap rejection and exact parity when timestamps are absent.
</objective>

  <action>
    Per 22-CONTEXT.md (Gap Segmentation, ALG-HRV-01):

    1. Add `#[serde(default)] pub rr_timestamps_s: Option<Vec<f64>>,` to the `HrvInput` struct after `input_ids` (metrics.rs:30). `#[serde(default)]` keeps JSON deserialisation backward compatible (`Option` defaults to `None`).

    2. Add a free function (place it near the `rmssd` helper, around metrics.rs:1895, NOT as a method, so it is unit-testable):
       `fn segment_rr_by_gaps(intervals: &[f64], timestamps: &[f64], gap_threshold_s: f64) -> Vec<Vec<f64>>`
       Behaviour:
       - If `timestamps.len() != intervals.len()` OR intervals is empty, return a single segment `vec![intervals.to_vec()]` (defensive — never panic, preserves today's behaviour).
       - Otherwise walk consecutive timestamps; whenever `timestamps[i] - timestamps[i-1] > gap_threshold_s`, close the current segment and start a new one. Push each interval into the current segment.
       - Return the Vec of segments (each a `Vec<f64>`).

    3. Add a free function `fn rmssd_segmented(segments: &[Vec<f64>]) -> f64` that:
       - Accumulates the sum of squared successive differences and the count of difference PAIRS across all segments (only within-segment `windows(2)` pairs).
       - Returns `(sum_sq / pair_count as f64).sqrt()` when `pair_count > 0`, else `0.0`.
       - This is the exact same math as the existing `rmssd` but summed across segments with a shared pair count — so a single segment reproduces `rmssd` bit-for-bit.

    Use a 3.0 constant for the gap threshold at the call site in Task 2; keep `gap_threshold_s` a parameter here for testability. Do NOT inline a fenced code block — name the signatures and behaviour only.

    Note for the executor: do NOT add a `Default` derive guess here; Task 3 handles existing-literal compilation explicitly.

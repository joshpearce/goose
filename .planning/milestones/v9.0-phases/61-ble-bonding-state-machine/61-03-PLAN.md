  <action>

  <what-built>
    A formal GooseBLEBondingManager now drives the connection portion of connectionState. Bond loss (CBError.peerRemovedPairingInformation / CBATTError.insufficientAuthentication) is detected on disconnect and transitions the manager to .cancelled("bond_lost") then .notStarted, re-entering the existing reconnect cycle. Bonding state (and the bonded device UUID when .completed) persists to UserDefaults under goose.swift.ble.bondingState / goose.swift.ble.bondingDeviceID and is reloaded on launch.
  </what-built>
  <how-to-verify>
    Bond-loss / BT-reset recovery (Success Criterion 2):
    1. Connect the app to the WHOOP device until the BLE status reaches "Ligado" / ready (this requires a real WHOOP on a real device; if only the simulator is available, perform the BT-toggle variant: toggle the Simulator's Bluetooth off and on while the app is running).
    2. Trigger bond loss: on a real device, toggle iOS Bluetooth off then on (Settings or Control Center), or reboot the device; this surfaces a disconnect with the bond-loss error code.
    3. Observe: the app log shows a `ble.bonding / bond.lost` warning, the bonding state returns to .notStarted, and the app re-enters the bonding/reconnect flow and reconnects WITHOUT any manual tap. The status should return to ready on its own.

    Persistence across app kill (Success Criterion 3):
    4. With the app connected (state .completed), force-quit the app (swipe up in the app switcher).
    5. Relaunch the app.
    6. Observe: on launch the bonding manager reads the persisted state — a previously .completed bond is restored as .completed(deviceID:) (the remembered device UUID is present), and the app resumes the reconnect flow toward the same device. Confirm goose.swift.ble.bondingState is "completed" in UserDefaults (via More > Debug if surfaced, or Xcode console).

    Observability (Success Criterion 1):
    7. Confirm the BLE/Device status surface reflects the bonding state through GooseAppModel (the existing connection status UI continues to show the correct state — it is now driven by the bonding manager).
  </how-to-verify>

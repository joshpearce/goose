<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OS scheduler → app | iOS BGAppRefreshTask wakes the app in background and invokes the launch handler on an arbitrary thread |
| app → BLE peripheral | background BLE scan/connect to the WHOOP band |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-60-03 | Denial of Service | handleBGAppRefresh crashes/returns without setTaskCompleted | mitigate | task.expirationHandler is set before any work and always calls setTaskCompleted(false); both connected and scan branches schedule a 20s asyncAfter that calls setTaskCompleted; scheduleNextBGAppRefresh runs first so the app keeps waking |
| T-60-04 | Denial of Service | BLE scan never stopped on background timeout (battery drain) | mitigate | The 20s asyncAfter timeout and the expirationHandler both call ble.stopScan() before completing the task |
| T-60-05 | Tampering | UserDefaults cooldown timestamp corrupted/missing | accept | A missing/invalid Date falls through to a sync (benign: at worst one extra fetch); the cast `as? Date` fails safe to nil; no crash and no security impact |
| T-60-06 | Information Disclosure | Background BLE fetch exposes data | accept | Reuses existing BLE pairing + GATT encryption model unchanged; no new data path; BGTask only triggers the existing sync command |
| T-60-SC | Tampering | npm/pip/cargo installs | accept | No external packages introduced; BackgroundTasks/Foundation/OSLog are system frameworks (RESEARCH Package Legitimacy Audit: not applicable) |
</threat_model>

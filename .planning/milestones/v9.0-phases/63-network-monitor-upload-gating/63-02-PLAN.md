<objective>
Gate the upload pipeline on network reachability and a non-empty APNs device token, add exponential-backoff retry (1s, 2s, 4s, max 60s) for 5xx server errors with a visible error state, and retry queued work automatically when connectivity returns.

Purpose: Satisfies NET-MON-01 success criteria 2, 3, and 4 — uploads fail visibly rather than silently, never fire offline, and require a registered device token. Matches WHOOP's WHPNetworkMonitor gating pattern.
Output: reachability gate in GooseAppModel+Upload, connectivity-return retry in GooseAppModel+Lifecycle, 5xx backoff + error state + token gate in GooseUploadService, and a minimal AppDelegate that registers for APNs and captures the device token into GooseAppModel.apnsDeviceToken.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app → Goose server (URLSession upload) | Outbound POST with Bearer token; server responses (status codes) are untrusted input driving retry logic |
| APNs → app | OS-delivered device token (trusted system origin); stored as opaque hex, never used as a secret beyond presence check |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-63-02 | Denial of service | Upload retry loop on persistent 5xx | mitigate | Exponential backoff capped at 60s per delay + bounded total attempts (≤6) so a failing server cannot cause a tight retry loop or battery drain |
| T-63-03 | Information disclosure | uploadErrorState surfaced to UI | mitigate | Error string contains only HTTP status code and a generic message — no token, URL, or payload contents |
| T-63-04 | Spoofing | APNs device token presence gate | accept | Token is used only as a non-empty presence check, not as an auth credential; server auth remains the existing Bearer token |
| T-63-SC | Tampering | npm/pip/cargo installs | accept | No package installs — UIKit/Network/Foundation are first-party OS frameworks (no new dependency per 63-CONTEXT) |
</threat_model>

# Phase 10.1 Context — BLE Main-Thread Publishing Fix

## Domain

Fix all `@Published` property mutations that happen on CoreBluetooth background threads in `GooseBLEClient+Commands.swift` and `GooseBLEClient+Parsing.swift`. These produce runtime warnings "Publishing changes from background threads is not allowed" when the app is connected to a WHOOP or HR monitor.

## Decisions

### Fix Strategy: Method-level `if !Thread.isMainThread` guard

Use the same guard pattern already present in `updateReconnectState`:

```swift
func updateXxx(_ value: ...) {
  if !Thread.isMainThread {
    DispatchQueue.main.async { [weak self] in self?.updateXxx(value) }
    return
  }
  // original body...
}
```

**Rationale:** Consistent with existing `updateReconnectState` pattern. Aligned with upstream PR #12 (Maloron — `DispatchQueue.main.async { [weak self] in guard let self else { return }; self.property = value }`). Fix is at the method level — zero impact on call sites.

**Rejected alternatives:**
- `@MainActor` on class — requires propagating through all callers (GooseAppModel, HealthDataStore, etc.), high blast radius
- Call-site wraps — duplicates logic, touches many files

### Audit Scope: Full audit of GooseBLEClient+Commands.swift + GooseBLEClient+Parsing.swift

Do NOT fix only the reported lines. Grep all `@Published` property mutations in both files and fix any that lack a main-thread guard. Confirmed reported violations:
- `updateConnectionState` line ~102: `connectionState = value` — no guard
- `updateActiveDeviceName` line ~111: `activeDeviceName = value` — no guard
- `updateReconnectState` line ~141: already has guard, but produces duplicate warning — investigate why
- `updateActiveDevice` (Parsing.swift ~430): `activeDeviceIdentifier = peripheral.identifier` — no guard

## Canonical Refs

- `GooseSwift/GooseBLEClient+Commands.swift` — primary fix target (updateConnectionState, updateActiveDeviceName, updateReconnectState, and any other @Published mutations)
- `GooseSwift/GooseBLEClient+Parsing.swift` — secondary fix target (updateActiveDevice + full audit)
- `GooseSwift/GooseBLEClient.swift` — @Published property list (lines 7-27)
- upstream PR #12 (b-nnett/goose) — reference pattern: `DispatchQueue.main.async { [weak self] in guard let self else { return }; self.property = value }`

## Code Context

**Existing guard pattern to replicate (GooseBLEClient+Commands.swift):**

```swift
func updateReconnectState(_ value: String) {
  if !Thread.isMainThread {
    DispatchQueue.main.async { [weak self] in self?.updateReconnectState(value) }
    return
  }
  // ... rest of body
}
```

**@Published properties on GooseBLEClient (GooseBLEClient.swift lines 7-27):**

All of these must only be mutated on the main thread. The fix ensures any method that mutates them has a guard.

## Success Criteria

1. No "Publishing changes from background threads is not allowed" runtime warnings when connected to WHOOP or HR monitor
2. `updateConnectionState`, `updateActiveDeviceName`, `updateActiveDevice`, and all other @Published-mutating methods in Commands.swift and Parsing.swift have the main-thread guard
3. The duplicate warning from `updateReconnectState` is resolved (investigate if the guard is correct or if a call site bypasses it)
4. No existing BLE behaviour or reconnect logic broken
5. `cargo test -p goose-core` passes (no Rust changes expected)

# Phase 10.1 Plan 01: BLE Main-Thread Publishing Fix Summary

Surgical application of method-entry `if !Thread.isMainThread` guards to every `@Published`-mutating method in `GooseBLEClient+Commands.swift` and `GooseBLEClient+Parsing.swift`, eliminating all "Publishing changes from background threads is not allowed" runtime warnings produced by CoreBluetooth delegate callbacks.

## What Was Built

Applied the identical guard pattern already present in `updateReconnectState` to 22 additional methods across two files. The guard is method-internal and idempotent — zero call-site changes, zero behavioural change.

## Duplicate updateReconnectState Warning — Root Cause

**Hypothesis confirmed:** The duplicate runtime warning attributed to `updateReconnectState` was not originating from `updateReconnectState` itself (which already had a guard). It originated from `updateConnectionState` — which had NO guard and was called on the same CoreBluetooth background queue call-stack (e.g., `connect(_:reason:)` calls `updateConnectionState("connecting")` followed by `updateReconnectState("connecting")`). Since `updateConnectionState` had no guard, it fired the warning. The guarded `updateReconnectState` call was silent. Observers attributed both warnings to the reconnect path, creating the illusion of a duplicate from `updateReconnectState`.

**Resolution:** Adding the guard to `updateConnectionState` (and all other unguarded mutators) eliminates the false duplicate. No special-casing was needed; no redundant guard was added to `updateReconnectState`.

## Return Value Decisions

| Method | Background-path return | Rationale |
|--------|------------------------|-----------|
| `validatedAlarmID(_:)` | `nil` | The only @Published mutation on the bg path is on the invalid-input branch (out-of-range). Returning nil is the same value the main-thread branch would return for that case. The re-dispatch performs the mutation asynchronously. |
| `handleStandardReadValue(_:characteristic:capturedAt:)` | `true` | Every characteristic UUID handled by this method (battery, device info) produces `true`. Returning `true` from the bg path tells the caller the characteristic was handled, preventing a redundant fallthrough. The actual @Published mutations are re-dispatched to main. The method is `@discardableResult` so callers that discard the value are unaffected. |

<objective>
Fix the Gen4 multi-notification frame reassembly bug (SYNC-09, issue #20).

Root cause: `handleHistoricalSyncValue` calls `frames(in: value)` with only the current notification's bytes. When a Gen4 type-47 body frame is larger than a single 512-byte BLE notification, `gen4Frames` breaks early on the first notification (incomplete frame) and discards the partial tail on the second notification because the continuation bytes do not start with `0xaa`. The result is zero complete type-47 frames written to SQLite during a Gen4 historical sync.

Fix: add a `gen4HistoricalFrameBuffer: Data` field to `GooseBLEHistoricalManager`. In `handleHistoricalSyncValue`, prepend any buffered bytes to the incoming notification value before calling `frames(in:)`. After the call, store any unconsumed tail back to the buffer. Clear the buffer on sync begin, complete, and fail.

Purpose: Gen4 historical sync writes complete type-47 packet bodies to SQLite instead of silently dropping them.
Output: Two modified Swift files; no Rust changes; issue #20 closed.
</objective>

    **Change 1 — handleHistoricalSyncValue (line 7-13 area):**

    The current implementation is:

        func handleHistoricalSyncValue(_ value: Data, characteristic: CBCharacteristic) {
          guard isHistoricalSyncing else {
            return
          }
          for frame in frames(in: value) {
            handleHistoricalSyncFrame(frame, characteristic: characteristic)

    Replace with the buffered version. The key semantics:
    - Prepend `historicalManager.gen4HistoricalFrameBuffer` to `value` to form `inputBytes`
    - Clear the buffer immediately (before the loop, optimistic clear)
    - Call `frames(in: inputBytes)` — note: `frames(in:)` dispatches to `gen4Frames` for Gen4 protocol
    - `gen4Frames` returns only complete frames; unconsumed tail bytes are not currently returned
    - To capture the unconsumed tail: after `frames(in: inputBytes)` returns, compute the tail as
      `inputBytes` minus the bytes consumed by all returned frames. Each Gen4 frame's total byte
      count is `4 + declaredLength` where `declaredLength = UInt16(frame[1]) | UInt16(frame[2]) << 8`
      and the 4-byte header is the Gen4 header. Store the unconsumed suffix to
      `historicalManager.gen4HistoricalFrameBuffer`.

    IMPORTANT: Examine the actual `gen4Frames` and `frames(in:)` implementations in
    `CoreBluetoothBLETransport+Parsing.swift` before writing the patch. There are two viable
    approaches:

    **Approach A (preferred — no API change):** Compute consumed bytes by summing frame sizes
    from the returned frames array, then store `inputBytes.suffix(from: consumedByteCount)` if
    non-empty and that suffix starts with a partial-frame indicator (i.e. it contains a `0xaa`
    at offset 0, meaning it is a frame start that could not complete). Store the tail only if
    the remaining bytes are a plausible partial frame (count > 0 and count < declared total
    length). Cap buffer at 8192 bytes to prevent unbounded growth on protocol errors.

    **Approach B (simpler but still correct):** Modify `gen4Frames` to return a `(frames: [Data], tail: Data)` tuple
    instead of `[Data]`, and update all call sites. Only use if Approach A proves error-prone
    after reading the parsing code.

    Use Approach A unless reading the parsing code reveals a compelling reason to use B.

    After determining the consumed count, store the suffix:

        let consumedCount = returnedFrames.reduce(0) { acc, frame in
          let declaredLength = frame.count >= 4
            ? Int(UInt16(frame[1]) | UInt16(frame[2]) << 8)
            : 0
          return acc + 4 + declaredLength
        }
        let tail = consumedCount < inputBytes.count
          ? Data(inputBytes[consumedCount...])
          : Data()
        historicalManager.gen4HistoricalFrameBuffer = tail.count > 0 && tail.count <= 8192
          ? tail
          : Data()

    Then iterate `returnedFrames` and call `handleHistoricalSyncFrame` per frame.

# Plan 108-01: Battery Level Wiring + UI (BAT-01)

## Objective

Verify and complete the end-to-end battery percentage pipeline for Gen4, Gen5,
and MG devices: confirm all three call sites (event-48, cmd-26, R22) correctly
invoke `applyBatteryLevel()`; confirm `DeviceView` on Home tab displays the
value; add missing Rust tests for `parse_event48_battery` and
`parse_cmd26_battery`; fix any gaps discovered during audit.

## Context

Research (108-RESEARCH.md) found that much more is already wired than the
phase context assumed. The three call sites are all present in
`GooseAppModel+NotificationPipeline.swift`. `DeviceView` already reads
`ble.batteryLevelPercent` and renders `BatteryRail`. The remaining work is:

1. Confirm `parse_event48_battery` Rust output includes `event48_battery_pct`
   key that `NotificationFrameParsing.swift` reads at line 118
2. Confirm `parse_cmd26_battery` Rust output is consumed correctly at
   `GooseBLEClient+BatteryCommands.swift:75`
3. Confirm cmd-26 is auto-sent on Gen4 connection (not only on explicit request)
4. Add missing Rust integration tests for battery parsing paths
5. Confirm build succeeds

<threat_model>
## Threat Model (ASVS L1)

| Asset | Threat | Disposition | Mitigation |
|-------|--------|-------------|------------|
| Battery level value | Integer overflow/underflow from malformed BLE payload | mitigate | `normalizedLevel = max(0, min(100, rawLevel))` clamp already in `applyBatteryLevel()` |
| Battery level value | Stale value shown after disconnect | mitigate | `batteryLevelPercent = nil` on disconnect at `CoreBluetoothBLETransport+Parsing.swift:507` |
| Cmd-26 auto-send | Sending command before BLE is ready | mitigate | `guard connectionState == "ready"` in `sendCmd26BatteryRequest()` |
| R22 battery byte | Out-of-range value (>100) displayed | mitigate | `batteryPct <= 100` guard at `NotificationPipeline.swift:663` |

No high-severity threats identified. All existing guards are sufficient.
</threat_model>

    Add a new private method `handleBodyLocationValue(_ value: Data, characteristic: CBCharacteristic)` in the HistoricalHandlers extension. Model it on the pattern of `handleClockCommandResponse` and `handleBatteryValue` (existing methods in this or the Commands extension).

    The method logic:
    - Guard that the characteristic UUID is in `notificationCharacteristicIDs` (or equivalent notification characteristic check used by other handlers in this extension — follow the existing pattern exactly).
    - Iterate over frames using `frames(in: value)` then extract payload using `payload(in: frame)`.
    - For each payload: guard `payload.count >= 9`. Guard that the first byte (`payload[0]`) matches the V5 commandResponse packet type (`V5PacketType.commandResponse` or `V5PacketType.puffinCommandResponse` — use both, as other handlers do). Guard that `payload[2] == 84` (the command byte).
    - Extract `location = Int(payload[6])` (payload[6] = body offset 1 = GarmentDeviceLocation value; body starts at payload[5] per the V5 commandResponse header: packetType[0] flags[1] commandByte[2] sequence[3] resultCode[4] body[5...]).
    - Map location to `Bool?`: `case 1` → true (WRIST); `case 2, 3, 4, 5, 7, 160` → false (known non-wrist); `default` → nil (UNKNOWN=0, NOT_CONCLUSIVE=128, unrecognised).
    - Dispatch to main queue and set `self?.isOnWrist = newValue`.
    - Call `record(source: "ble", title: "body_location.response", body: "location=\(location) confidence=\(payload[7]) status=\(payload[8]) isOnWrist=\(String(describing: newValue))")`.

    IMPORTANT — pitfall to avoid: do NOT route this through `handleSensorStreamValue`. The cmd 0x54 response is a commandResponse frame; `handleSensorStreamValue` uses a dictionary allow-list keyed on sensor command kinds that does not include 84, so 0x54 responses would be silently discarded there (Pitfall 3 in RESEARCH.md).

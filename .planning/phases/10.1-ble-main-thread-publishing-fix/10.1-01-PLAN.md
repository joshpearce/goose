<objective>
Eliminate all SwiftUI "Publishing changes from background threads is not allowed" runtime warnings emitted by `GooseBLEClient` when CoreBluetooth delegate callbacks (which run on `coreBluetoothQueue`, a background queue) mutate `@Published` properties.

The fix is a surgical, method-level main-thread guard applied to every method in `GooseBLEClient+Commands.swift` and `GooseBLEClient+Parsing.swift` that mutates an `@Published` property and can be reached from a background thread. The guard pattern is identical to the one already present in `updateReconnectState` (per D-01) and is consistent with upstream PR #12. The guard is method-internal and idempotent — zero call-site changes, zero behavioural change.

Purpose: Removing these warnings makes BLE diagnostics readable, prevents undefined SwiftUI publish-ordering, and closes a known data-race risk (BT queue vs. main thread) flagged in STATE.md.
Output: Two edited Swift files with main-thread guards on every affected method, plus a resolved duplicate `updateReconnectState` warning.
</objective>

<reference_pattern>
The locked fix strategy (D-01) replicates the existing guard in updateReconnectState
(GooseBLEClient+Commands.swift, ~line 139):

  func updateXxx(_ value: ...) {
    if !Thread.isMainThread {
      DispatchQueue.main.async { [weak self] in self?.updateXxx(value) }
      return
    }
    // original body, unchanged
  }

Rules for applying the guard:
- The re-dispatched call MUST forward the EXACT same arguments to the SAME method (recursive re-entry on the main thread).
- The guard goes at the very top of the method body, before any other statement.
- The original method body is left byte-for-byte unchanged below the guard.
- For methods with multiple parameters, forward every parameter in the recursive call, preserving argument labels.
- For methods with NO parameters (e.g. updateRememberedDeviceDescription()), the recursive call is self?.updateRememberedDeviceDescription().
- For methods with default-valued parameters (e.g. clearRememberedDevice(reason:source:)), forward the actual received values, not the defaults.
- Methods that mutate @Published ONLY by calling an already-guarded helper (updateConnectionState / updateReconnectState / updateActiveDeviceName) do NOT need their own guard. Add a guard ONLY to methods that mutate an @Published property DIRECTLY.
- For methods that return a value (e.g. validatedAlarmID returns UInt8?), the background-thread early-return path re-dispatches the mutation AND returns the safe default (nil); document the chosen return in the SUMMARY.
- Do NOT add a guard to static functions (they cannot touch instance @Published state) or to pure parsing/encoding helpers.
</reference_pattern>

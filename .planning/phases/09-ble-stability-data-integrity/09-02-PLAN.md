<objective>
Fix CR-02 (FIX-01): propagate a per-session device id from the Swift bridge call into the `capture_sessions.active_device_id` column so HR monitor frames are no longer stored with a NULL device id.

Purpose: Today the batch import path hardcodes `active_device_id: None` (capture_import.rs line 400), so every live-captured session row has a NULL device id and there is no way to trace which physical device produced the data. Per D-01/D-02 the fix is on the INSERT side and is session-scoped (NOT a per-frame field on CapturedFrameInput, per RESEARCH Pitfall 5). The Swift side (Plan 03) will supply `peripheral.identifier.uuidString`; this plan makes the Rust core accept and store it.

Output: a new optional `active_device_id` field on the batch import args, threaded through `CapturedFrameBatchOptions` into the `start_capture_session` call, plus a test proving the stored value is non-NULL.

Depends on Plan 09-01 because both edit `bridge.rs`; sequencing avoids edit conflicts and lets the storage/panic work land first.
</objective>

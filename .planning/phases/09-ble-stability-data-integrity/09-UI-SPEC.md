## Component Inventory

All additions are to `ConnectionContentView` inside `ConnectionView.swift`. No new files for UI.

### 1. Status Section — Reconnect Attempt Counter Row

**Location:** `Section("Status")` — add after existing `LabeledContent("Reconnect", value: ble.reconnectState)`.

**Component:** `LabeledContent` (existing pattern).

**New `@Published` property required on `GooseBLEClient`:**
```
@Published var reconnectState = "idle"   // already exists — update format per D-10 Specifics
```

The existing `reconnectState` string is updated during backoff to include the attempt count:
- Idle: `"idle"`
- Reconnecting: `"reconnecting (attempt 3/10)"`
- Failed: `"failed after 10 attempts"`

No new `LabeledContent` row is needed — the existing "Reconnect" row carries the attempt count in its value string. This is consistent with the existing `historicalSyncValue` pattern (state + metadata in one string).

### 2. Actions Section — Retry and Stop Buttons

**Location:** `Section("Actions")` — add after existing "Reconnect Remembered" button.

**Components:** Two `Button` views, conditionally shown.

**Visibility rules:**
- "Retry Reconnect" button: shown when `ble.reconnectState` starts with `"failed"` (i.e., after 10 attempts). Tapping restarts backoff from attempt 1.
- "Stop Reconnecting" button: shown when `ble.reconnectState` contains `"reconnecting"` (i.e., backoff cycle is active). Tapping aborts the cycle and sets state to `"idle"`.
- Both hidden when `reconnectState == "idle"` or `reconnectState == "disconnected"`.

**Button markup:**
```swift
if ble.isReconnecting {
  Button("Stop Reconnecting") {
    ble.stopReconnect()
  }
}
if ble.reconnectFailed {
  Button("Try Again") {
    ble.retryReconnect()
  }
  Text("Reconnection failed after 10 attempts. Tap \"Try Again\" to restart.")
    .font(.caption)
    .foregroundStyle(.secondary)
}
```

**New `@Published` computed helpers required on `GooseBLEClient`:**
- `var isReconnecting: Bool` — true when backoff cycle is active
- `var reconnectFailed: Bool` — true when circuit breaker tripped (10 attempts exhausted)

Alternatively, both can be derived from `reconnectState` at the view layer without new properties — executor may choose either approach.

### 3. Event Log Section — Storage Compaction Result

**Location:** `Section("Event Log")` — populated automatically via `ble.record()`.

**Trigger (from D-10):** Called only when `compacted_rows > 0`. Silent otherwise.

**Format:** Single `ble.record()` call with:
- `source`: `"storage"`
- `title`: `"Storage compacted"`
- `body`: `"1 200 rows, 18 MB freed"` (formatted with actual values)

This entry surfaces in the Event Log via the existing `GooseMessageStore` + `ForEach(messageStore.messages)` pattern. No new UI component. Typography matches the existing Event Log row layout (caption timestamp/source, subheadline title, monospaced caption body).

## Interaction States

### Reconnect Backoff Cycle

| State | UI Representation | `reconnectState` value |
|-------|------------------|----------------------|
| No reconnect | "Reconnect" row shows `"idle"` | `"idle"` |
| Attempt in progress | "Reconnect" row shows attempt N | `"reconnecting (attempt 3/10)"` |
| Attempt in progress | "Stop Reconnecting" button visible in Actions section | same |
| Circuit breaker tripped | "Reconnect" row shows `"failed after 10 attempts"` | `"failed after 10 attempts"` |
| Circuit breaker tripped | Failure message + "Try Again" button visible in Actions section | same |
| User taps Stop | State returns to `"idle"`, buttons hidden | `"idle"` |
| User taps Try Again | Cycle restarts from attempt 1 | `"reconnecting (attempt 1/10)"` |

**No modal alerts, no confirmation dialogs.** All state is reflected inline in the existing `List` layout.

### Storage Compaction

| Trigger | UI Effect |
|---------|-----------|
| `compacted_rows == 0` | Silent — nothing appears in Event Log |
| `compacted_rows > 0` | Single `ble.record()` entry appears at top of Event Log with title "Storage compacted" |

**No progress indicator, no loading spinner.** Compaction is fast (SQLite UPDATE with LIMIT) and runs synchronously on the background queue before the bridge call returns. Log entry appears after completion.

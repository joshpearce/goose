<interface_context>
Current AppShell signature passes StateFlow<T> down to child screens (HomeScreen, HealthScreen, MoreScreen),
which each call collectAsStateWithLifecycle(). connectionState is already a resolved value.
MainActivity already collects connectionState with collectAsStateWithLifecycle() but passes the other five
flows (liveHeartRateBPM, recoveryScore, strainScore, sleepScore, serverUrl) raw.

This plan moves all lifecycle-aware collection up into MainActivity. After this plan:
- AppShell receives resolved values (Int?, Float?, String) instead of StateFlow<T> for those fields,
  plus the new resolved uploadStatus value.
- HomeScreen / HealthScreen / MoreScreen signatures change correspondingly from StateFlow<T> to the
  resolved value type; MoreScreen drops its own collectAsStateWithLifecycle() for serverUrl.

New public AppViewModel surface consumed by MainActivity:
- `val uploadStatus: StateFlow<UploadState>` (UploadState sealed class defined in AppViewModel.kt or a sibling file).

GooseBridge.safeHandle() returns ok:false JSON on failure and never throws — callers branch on `ok`.
WhoopBleClient.syncCompleteEvent: SharedFlow<Unit> (introduced in plan 128-01) replaces onSyncComplete;
AppViewModel collects it in viewModelScope.
</interface_context>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ViewModel -> Compose UI | StateFlows cross into the Compose runtime; incorrect collection drops updates |
| UI -> BLE client | Public bleClient lets the UI bypass ViewModel lifecycle guards |
| App -> self-hosted server | Upload HTTP I/O crosses the network boundary; failures must be visible |

| T-128-05 | Information disclosure (loss) | queryScore swallowing bridge errors | medium | mitigate | Log.e on exception and ok:false (Task 2) |
| T-128-06 | Information disclosure (loss) | invisible upload failures | medium | mitigate | Observable UploadState surfaced in MoreScreen (Task 2) |

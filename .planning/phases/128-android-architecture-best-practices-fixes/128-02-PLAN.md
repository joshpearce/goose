Fix the four Compose/ViewModel/error-handling findings from the Phase 127 audit (A-04 StateFlow collection, A-05 bleClient visibility, A-06 queryScore swallowing, A-08 observable upload state) without changing app behaviour.

</objective>

<interface_context>
Current Compose wiring (verified during planning):
- MainActivity.onCreate collects only connectionState with collectAsStateWithLifecycle() (line 21);
  the other five fields (liveHeartRateBPM, recoveryScore, strainScore, sleepScore, serverUrl) are passed
  to AppShell as raw StateFlow<T> (lines 22-26).
- AppShell signature (lines 24-32) declares liveHeartRateBPM/recoveryScore/strainScore/sleepScore as
  StateFlow<T> and serverUrl as StateFlow<String>; it forwards these StateFlows to HomeScreen
  (line 66, liveHeartRateBPM: StateFlow<Int?>), HealthScreen (lines 71-76, three StateFlow<Float?>),
  and MoreScreen (line 80, serverUrl: StateFlow<String>).
- HomeScreen (line 22) collects liveHeartRateBPM itself; HealthScreen (lines 25-27) collects its three
  flows itself; MoreScreen (line 23) collects serverUrl itself.

This plan moves ALL lifecycle-aware collection up into MainActivity. After this plan:
- AppShell receives resolved values (Int?, Float?, String, and the new UploadState) instead of
  StateFlow<T>; zero StateFlow<T> parameters remain in AppShell or any child screen signature.
- HomeScreen / HealthScreen / MoreScreen signatures change from StateFlow<T> to the resolved value type;
  each drops its own collectAsStateWithLifecycle() call.

Interface added by plan 128-01 (already landed in wave 1):
- WhoopBleClient.syncCompleteEvent: SharedFlow<Unit> replaces onSyncComplete; AppViewModel collects it
  in viewModelScope.

New public AppViewModel surface consumed by MainActivity:
- `val uploadStatus: StateFlow<UploadState>` (UploadState sealed class defined in AppViewModel.kt or a
  sibling UploadState.kt).

GooseUploadClient is a singleton `object` (object GooseUploadClient). Per cross-AI review MEDIUM, it
exposes a companion-level `val uploadState: MutableStateFlow<UploadState>` directly on the object (the
simplest mechanism — no refactor to a class). AppViewModel reads/republishes it. The process-global
nature is acceptable for a single-upload-at-a-time client.

GooseBridge.safeHandle() returns ok:false JSON on failure and never throws — callers branch on `ok`.
</interface_context>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ViewModel -> Compose UI | StateFlows cross into the Compose runtime; incorrect collection drops updates |
| UI -> BLE client | Public bleClient lets the UI bypass ViewModel lifecycle guards |
| App -> self-hosted server | Upload HTTP I/O crosses the network boundary; failures must be visible |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-128-04 | Tampering | UI bypassing ViewModel via public bleClient | medium | mitigate | Make bleClient private; route access through ViewModel (Task 1) |
| T-128-05 | Information disclosure (loss) | queryScore swallowing bridge errors | medium | mitigate | Log.e on exception and ok:false (Task 2) |
| T-128-06 | Information disclosure (loss) | invisible upload failures | medium | mitigate | Observable UploadState surfaced in MoreScreen (Task 2) |
| T-128-SC | Tampering | npm/pip/cargo installs | low | accept | No package installs in this plan (Kotlin source only) |
</threat_model>

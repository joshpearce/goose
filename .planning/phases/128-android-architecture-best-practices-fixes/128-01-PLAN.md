<interface_context>
Interface change consumed by plan 128-02 (AppViewModel), which runs AFTER this plan in wave 2:
- REMOVE public property: `var onSyncComplete: (() -> Unit)?`
- ADD public property: `val syncCompleteEvent: SharedFlow<Unit>` (backed by a private MutableSharedFlow)
- The MutableSharedFlow is configured per D-07 specifics: `MutableSharedFlow<Unit>(replay = 0, extraBufferCapacity = 1, onBufferOverflow = BufferOverflow.DROP_OLDEST)`.
- completeSyncIfActive() emits to syncCompleteEvent via tryEmit(Unit) (non-suspending; safe from the BLE callback thread) instead of invoking onSyncComplete.

GooseBridge.safeHandle(request: String): String already returns an error-shaped JSON object
`{"ok":false,"result":null,"error":{"message":"..."},"timing":null}` instead of throwing.
Callers must parse the response and branch on the `ok` boolean — they must NOT assume success.

Concurrency note (resolved from cross-AI review HIGH-2): the BLE callback methods
(onConnectionStateChange, onCharacteristicWrite, onCharacteristicChanged) run on the Android BLE
thread and are NOT suspend functions. kotlinx.coroutines Mutex.withLock is a suspend function and
MUST NOT be called from these callbacks (it would not compile, and runBlocking would deadlock the
BLE thread). Therefore sync state is protected with java.util.concurrent.atomic primitives
(AtomicBoolean / AtomicReference), which provide lock-free atomic check-and-set safe from any thread.
</interface_context>

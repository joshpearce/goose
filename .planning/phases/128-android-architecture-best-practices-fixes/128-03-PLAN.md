<interface_context>
Current build facts (verified during planning):
- Kotlin 2.4.0, AGP 9.2.0, compileSdk 36, minSdk 26 (version catalog at android/gradle/libs.versions.toml).
- The app module applies ONLY `android.application` + `kotlin.compose` plugins. There is NO
  `org.jetbrains.kotlin.android` plugin applied, and no `kotlin-android` alias in the version catalog.
  KSP (and therefore Hilt annotation processing) REQUIRES the kotlin-android plugin to be applied first
  (cross-AI review HIGH-5). Without it the ksp configuration is not recognised.
- No Hilt and no KSP present today. Root build.gradle.kts only declares android.application +
  kotlin.compose (apply false).
- No Application subclass exists; the manifest <application> tag has no android:name.
- AppViewModel currently constructs `MetricsViewModel(app)` and `SettingsViewModel(app)` directly as
  PRIVATE fields (lines 28-29) — this is already the D-09a target state for encapsulation.
- MetricsViewModel and SettingsViewModel both extend AndroidViewModel(app).

Why D-09a is primary (cross-AI review HIGH-6 / HIGH-7):
- KSP releases are pinned to exact Kotlin versions. A KSP release targeting Kotlin 2.4.0 may not exist at
  execution time; the original plan left the version as an unresolvable placeholder "2.4.0-<kspPatch>".
- The original plan's `@HiltViewModel + @Inject constructor(app: Application) : AndroidViewModel(app)`
  pattern is incorrect for Hilt; the correct pattern drops AndroidViewModel and uses
  `@Inject constructor(@ApplicationContext context: Context) : ViewModel()`.
- Given two toolchain blockers and one incorrect pattern, attempting Hilt by default would most likely
  fail CI. D-09a (private sub-ViewModels + documented deferral) delivers the encapsulation value safely.
</interface_context>

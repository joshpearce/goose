<objective>
Port the WHOOP historical sync pipeline to Android. Add `startHistoricalSync()` to
`WhoopBleClient`, build the GET_DATA_RANGE and SEND_HISTORICAL_DATA command frames with
correct byte layouts, route type-47 body frames through the existing FrameReassembler →
GooseBridge pipeline, auto-trigger on BLE connect, and verify that at least one
Gen5 historical sync produces `decoded_frames` rows in SQLite.
</objective>

<context>
iOS parity target confirmed from `GooseSwift/CoreBluetoothBLETransport.swift`:
- GET_DATA_RANGE = command byte 34 (0x22)
- SEND_HISTORICAL_DATA = command byte 22 (0x16)
- HISTORICAL_DATA_RESULT ack = command byte 23 (0x17)

Command frame wire format (from iOS `buildCommandFrame` + `HistoricalCommandKind.payload`):
  [0x01, bodyLenLow, bodyLenHigh, outerSeq, innerSeq, commandByte, ...data]

Gen5 payload for GET_DATA_RANGE and SEND_HISTORICAL_DATA: standard `kind.payload` (non-zero).
Gen4 payload override: [0x00] (single zero byte) — `usesPageSequenceSync` path.

Notification routing: historical data frames arrive on the same characteristics as live
data. During active sync, all frames are passed to `capture.import_frame_batch` with
`source="historical_sync"`. SYNC-08 routing fix (Phase 98) in Rust core handles correct
internal routing — Android just passes bytes through.

`capture.import_frame_batch` bridge call already exists in `WhoopBleClient.buildImportRequest()`.
Reuse it with `source="historical_sync"` for historical frames.

`FrameReassembler.kt` (Phase 104) handles Gen4 multi-notification reassembly.
`WhoopBleClient.kt` (Phase 104) has all GATT plumbing needed.
</context>

<threat_model>
- T-105-01: Concurrent sync prevention — `syncInProgress` flag checked before writing commands; multiple BLE connect events do not spawn parallel syncs
- T-105-02: BLE callback thread blocking — bridge call dispatched to `Dispatchers.IO`; never called on BLE callback thread
- T-105-03: Command write sequencing — next command only written from `onCharacteristicWrite` callback; back-to-back writes without confirmation are not issued
</threat_model>

  private fun buildCommandFrame(sequence: Byte, command: Byte, data: ByteArray): ByteArray {
      val body = byteArrayOf(sequence, command) + data
      val bodyLen = body.size
      return byteArrayOf(
          PACKET_TYPE_COMMAND,
          (bodyLen and 0xFF).toByte(),
          ((bodyLen ushr 8) and 0xFF).toByte(),
          sequence,
      ) + body
  }

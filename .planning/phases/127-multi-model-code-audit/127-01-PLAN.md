<scope_reference>
This phase is ANALYSIS ONLY. No source files are modified. `files_modified` is empty by design; all output is planning artifacts.

Audit scope (per D-03):
- **Rust**: all 61 non-bin `.rs` files under `Rust/core/src/` — priority focus: `bridge/`, `store/`, `android_jni.rs`, `metric_features.rs`, `historical_sync.rs`, `protocol.rs`
- **Android Kotlin**: all 17 `.kt` files under `android/app/src/main/` — priority focus: `bridge/GooseBridge.kt`, `ble/WhoopBleClient.kt`, `viewmodel/AppViewModel.kt`, `viewmodel/MetricsViewModel.kt`
- **iOS Swift**: `GooseAppModel*.swift`, `GooseRustBridge.swift`, `GooseBLEClient*.swift`, `HealthDataStore*.swift`, `NotificationFrameParsing.swift`, `CaptureFrameWriteQueue.swift`, `OvernightSQLiteMirrorQueue.swift` (36 files)

Required analysis axes (per D-08), used identically by all three models and as the consolidation structure:
1. Module organisation — god files, single-responsibility violations, module boundary clarity
2. JNI/FFI patterns — error propagation, SAFETY contracts, lifecycle assumptions
3. Threading/concurrency — coroutine scope (Android), DispatchQueue / @MainActor safety (Swift), Rust Send/Sync
4. Null-safety / error handling — silent discards, `unwrap()` equivalents, `try?` silent failures, Kotlin `!!` operators
5. Android-specific — Compose state management, ViewModel ownership, lifecycle binding
</scope_reference>

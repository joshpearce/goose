# Plan 109-01: MG Sync Logging Hardening + Detection Gate + Issue #22

## Objective

Fix `DeviceCatalog` to correctly identify WHOOP MG in `generationLabel` and `historicalDeviceType`.
Strengthen the hardware gate annotation in `CoreBluetoothBLETransport+Commands.swift`. Post a
neutral progress comment on GitHub issue #22. Verify build succeeds.

Root cause (from research): MG gets the Gen5 sync path correctly (no routing bug), but
`DeviceCatalog.generationLabel` returns `"gen5"` instead of `"mg"` and `historicalDeviceType`
returns `"GOOSE"` instead of `"WHOOP_MG"` — both missing an MG branch. The hardware gate
annotation in Commands.swift needs a `hardware_gate:` suffix explaining the testing limitation.

## Context

- WHOOP MG (`deviceKind = "WHOOP_MG"`) follows the Gen5 `stream` historical sync path — correct
- `DeviceCatalog.generationLabel` at lines 25–28: only checks `wireProtocol == .gen4`; returns `"gen5"` for MG
- `DeviceCatalog.historicalDeviceType` at line 39: returns `capabilities?.wireProtocol.bridgeString ?? "GOOSE"` — gen5 bridgeString is `"GOOSE"`, so MG logs as `"GOOSE"`
- `DeviceCatalog.displayGeneration` already correctly returns `"MG"` for WHOOP_MG — do not change
- No changes to HistoricalHandlers, HistoricalCommands, GooseBLEHistoricalManager, or Rust
